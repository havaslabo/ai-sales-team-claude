# 稟議支援スキル

日本BtoB営業の最大のボトルネックである「稟議プロセス」を突破するための支援パッケージを生成します。営業担当者（チャンピオン）が社内の上長・役員・購買委員会に承認を取るための資料一式を自動生成します。

日本の稟議プロセスにおける重要な前提：
- 稟議は「担当者が通す」のではなく「担当者が上に説明しやすい状態を作る」プロセスです。
- 意思決定者は多忙です。5分で判断できる資料が最も通りやすい。
- リスクを隠すと後から信頼を失います。デメリットも正直に書く方が稟議は通りやすい。
- 数値の根拠が明示されていない資料は差し戻されます。出典・算出根拠を必ず明記してください。

## 呼び出し方法

```
/sales ringi <prospect_or_analysis_file>
```

`<prospect_or_analysis_file>` は見込み客の分析ファイル（PROSPECT-ANALYSIS.md 等）のパス、または企業名・製品名・課題を直接入力します。

---

## ステップ1：インプットの収集

以下の情報を収集します。既存の分析ファイルが存在する場合は自動的に読み込みます。不足している情報のみ確認してください。

### 必須インプット

1. **見込み客企業名**：正式名称・業種・規模（従業員数・売上規模）
2. **提案製品・サービス名**：導入を検討している製品・サービスの名称
3. **導入目的と解決したい課題**：現状の問題点、なぜ今これが必要か
4. **費用（円建て・税込）**：初期費用・月額/年額・保守費用・導入支援費用
5. **期待効果**：定量効果（〇〇%削減、月〇時間削減）と定性効果
6. **稟議の承認者**：決裁者（役職）、承認フロー（例：担当→課長→部長→役員）
7. **稟議提出予定日と希望導入時期**

### オプションインプット（品質向上に貢献）

8. **競合比較情報**：他社製品・現状維持・内製化との比較検討状況
9. **セキュリティ・コンプライアンス要件**：情報システム部門の確認事項
10. **導入実績・事例**：同業種・同規模の導入事例（定量実績付き）
11. **担当者（チャンピオン）情報**：申請部署・申請者氏名・役職

### 既存分析ファイルの自動取り込み

作業ディレクトリ内の以下のファイルが存在する場合は自動的に読み込みます。

- **PROSPECT-ANALYSIS.md**：企業概要、課題、機会評価を抽出
- **LEAD-QUALIFICATION.md**：BANT/MEDDIC の資格確認データ、予算・意思決定情報を抽出
- **COMPETITIVE-INTEL.md**：競合ポジショニング、現行ソリューション情報を抽出
- **DECISION-MAKERS.md**：承認者・意思決定構造を抽出
- **OBJECTION-PLAYBOOK.md**：想定反論と回答を FAQ セクションに活用

## ステップ2：稟議支援パッケージの生成

以下の5つのドキュメントで構成される「RINGI-SUPPORT.md」を生成します。すべて日本語・敬体（です・ます調）で統一します。

### 執筆原則（全セクションに適用）

1. **担当者目線で書く。** 担当者が「上に説明しやすい」言葉・構成にします。専門用語は避け、役員が読んでも理解できる平易な表現を使います。
2. **数値は具体的に。** 「大幅削減」ではなく「月40時間・年間480時間の工数削減」のように書きます。算出根拠も明示します。
3. **リスクを隠さない。** デメリット・リスクも正直に記載します。対策とセットで書くことで信頼性が増します。
4. **今やるべき理由を示す。** 「なぜ今このタイミングか」を必ず明示します（年度末予算消化・競合導入・規制対応等）。
5. **稟議が差し戻されないように作る。** 想定される質問・懸念点をあらかじめ文書内で回答します。

---

#### 構成

（2〜3行で簡潔に）

### ドキュメント2：稟議書用FAQ（よくある質問と回答）

上長・役員・購買委員会からの **想定質問に対する回答集** を生成します。担当者が口頭で説明するときにも使えるよう、回答は完結で説得力のある内容にします。

#### 必ず含める質問項目

**Q1. なぜ今このタイミングで導入する必要があるのか？**
- 緊急性の根拠（競合他社の動向・規制対応・現行システムの限界・年度予算等）を具体的に記載

**Q2. 他社製品と比較検討したか？なぜこの製品を選んだのか？**
- 比較検討の実施を明示し、選定理由を3点以内で簡潔に記載

**Q3. セキュリティ・情報管理は大丈夫か？**
- 認定・認証（ISO27001、SOC2 等）、データ保管場所（国内/海外）、アクセス制御を明記

**Q4. 費用対効果（ROI）は明確か？**
- 投資額と回収見込みを数値で示す。算出根拠も1行で添付

**Q5. 導入後のサポート体制はどうなっているか？**
- サポート時間帯・窓口・SLA・担当者（カスタマーサクセス等）を明記

**Q6. 契約を解約した場合のリスクは何か？**
- 解約条件・データ移行・費用・代替手段を正直に記載

**Q7. 導入に失敗した場合はどうなるか？**
- 失敗リスクの低減策（試験導入・段階的展開・ベンダーサポート）を記載

**Q8. 社内の業務負荷はどれくらいか？**
- 導入担当者の工数・期間・必要なスキルセットを明記

### ドキュメント3：比較検討表

**競合他社 / 現状維持 / 内製開発** の3択比較表を生成します。意思決定者が選択肢を客観的に比較できる形式にします。

#### 評価軸と比較表フォーマット

```
| 評価軸           | [提案製品名]       | [競合A]            | 現状維持           | 内製開発           |
|------------------|--------------------|--------------------|--------------------|--------------------|
| 主要機能         |                    |                    |                    |                    |
| 初年度コスト     |                    |                    |                    |                    |
| 月額コスト       |                    |                    |                    |                    |
| 導入期間         |                    |                    |                    |                    |
| サポート体制     |                    |                    |                    |                    |
| セキュリティ認証 |                    |                    |                    |                    |
| 国内実績         |                    |                    |                    |                    |
| カスタマイズ性   |                    |                    |                    |                    |
| 総合評価         | ★★★★★            | ★★★☆☆            | ★★☆☆☆            | ★☆☆☆☆            |
```

#### 比較表の執筆ルール

- 競合他社名を具体的に記載できる場合は記載する（不明な場合は「競合A」「競合B」）
- 「現状維持」のデメリットは必ず定量化する（「対応コスト増加：年間〇〇円相当」等）
- 内製開発のコストは人件費・期間・リスクも含めて試算する
- 提案製品を過大評価しない（デメリットも正直に記載する）

### ドキュメント4：導入スケジュール

**稟議申請から本番運用開始まで** のガントチャート風スケジュールを生成します。

#### フォーマット

```
■ 導入スケジュール概要（〇ヶ月計画）

フェーズ            | 内容                        | 期間          | 担当
--------------------|-----------------------------|---------------------------------
1. 稟議・契約       | 稟議申請・承認・契約締結     | 〇月〜〇月    | 申請部署・購買部
2. キックオフ       | 要件定義・体制構築           | 〇月          | ベンダー・情報システム
3. 環境構築         | システム設定・データ移行     | 〇月〜〇月    | ベンダー・担当者
4. 試験導入         | パイロット運用・フィードバック| 〇月〜〇月    | 担当部署・ベンダー
5. 研修・定着化     | 全社展開・操作研修           | 〇月          | ベンダー・担当者
6. 本番運用開始     | 正式稼働・効果測定開始       | 〇月          | 全関係者

■ 重要マイルストーン
・〇月〇日：稟議承認期限
・〇月〇日：契約締結期限（導入スケジュールに間に合わせるため）
・〇月〇日：本番稼働開始
・〇月〇日：初回効果測定レポート
```

## ステップ3：ファイルの保存

生成した稟議支援パッケージを以下のファイル名で保存します。

```
RINGI-SUPPORT.md
```

保存後、以下を担当者に伝えます。
- 生成したパッケージの概要（含まれるドキュメント一覧）
- カスタマイズが必要な箇所（数値・日付・承認者名等のプレースホルダー）
- 稟議提出前に確認すべき事項（費用の税込確認・承認フローの確認等）

---

## 重要なルール

- **担当者目線で書く。** 担当者が「上に説明しやすい」資料を作る。役員が自社でも読める平易な表現を使う。
- **数値は具体的に。** 「〇〇%削減」「月〇時間の工数削減」「年間〇〇円のコスト削減」のように具体化する。算出根拠も添付する。
- **リスクを隠さない。** デメリットも正直に書く方が信頼される。対策とセットで記載する。
- **日本語・敬体（です・ます調）で統一する。** 稟議書は社内公式文書のため、文体を一貫させる。
- **RINGI-SUPPORT.md に保存する。** ファイル名は固定。
