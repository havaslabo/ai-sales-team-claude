# PoC（試験導入）提案スキル

## 呼び出し方

```
/sales poc <company_name>
```

## 入力

- 企業名
- 課題内容
- 製品 / サービス名
- 提案期間

## 出力

`POC-PROPOSAL-[会社名].md`

---

## 目的

日本のBtoB営業では「まず試してから本番導入を判断したい」という要望が多い。PoC提案はエンタープライズ商談の鍵。いきなり本番契約を提案するのではなく、PoC → 評価 → 本番という段階的アプローチが日本では有効。

---

## 生成するPoC提案書の構成

### 1. 表紙
- 提出先（企業名・部門名・担当者名）
- 提出日
- 提出者（自社名・担当者名・連絡先）
- 文書の機密区分（例：「社外秘」「取扱注意」）
- 文書タイトル（例：「[製品名] 試験導入（PoC）提案書」）

### 2. PoCの目的・背景
- お客様が抱える課題の認識を整理（ヒアリング内容をもとに記載）
- 現状の問題点・業務への影響
- PoCを通じて検証・解決したいこと
- 本番導入に向けた判断材料としてのPoC位置づけ

### 3. PoCスコープ（対象範囲）
- 対象部署・チーム
- 対象ユーザー数（人数・役職）
- 検証する機能・シナリオ（具体的に列挙）
- 除外する機能・範囲（スコープ外の明示）
- 利用環境（本番環境 / 検証環境 / テナント分離の有無）

### 4. 成功基準（KPI）
- PoCを「成功」と判定するための定量的な指標を定義する
  - 例：ユーザー満足度スコア 4.0/5.0 以上
  - 例：対象業務の処理時間を現状比30%削減
  - 例：週次アクティブユーザー率 80%以上
- 最終評価の実施方法（アンケート・ログ分析・ヒアリング等）
- 判定基準の合意（PoC開始前に双方で合意した内容を明記）

### 5. PoC実施計画
- **期間**（推奨：1〜3ヶ月）
- **週次スケジュール**
  - Week 1：環境構築・初期設定・キックオフ
  - Week 2〜N-1：検証・フィードバック収集・改善対応
  - Week N：最終評価・まとめ・本番移行判断
- **役割分担**

  | 役割 | お客様側 | 提供者側 |
  |------|----------|----------|
  | 推進責任者 | PoC担当マネージャー | 担当営業 / CSM |
  | 技術窓口 | 社内IT担当 | テクニカルサポート |
  | 現場利用者 | 対象部署ユーザー | — |
  | 評価責任者 | 導入判断権限者 | — |

- **定例ミーティング**（推奨：隔週30分）
  - 進捗確認・課題共有・次週アクション

### 6. PoC費用
- 無償 / 有償の別と条件を明示
- 無償の場合：条件（期間・ユーザー数・機能制限等）
- 有償の場合：費用感と根拠
- 本番移行時の費用見込み（参考値）
- 無償PoC後に本番契約が成立しなかった場合の扱い

### 7. 本番移行の条件と見込み
- 成功基準達成を前提とした本番移行の想定スケジュール
- 本番導入の範囲（PoC対象部署から全社展開まで）
- 本番契約の形態（ライセンス数・プラン・契約期間）
- 移行作業の概要（データ移行・設定引き継ぎ等）

### 8. サポート体制
- PoC期間中の問い合わせ窓口（担当者・メール・電話）
- サポート対応時間（平日9:00〜18:00等）
- 障害・緊急時の対応フロー
- オンボーディング支援（初期設定サポート・操作説明会等）

### 9. お問い合わせ先
- 担当営業の氏名・役職・連絡先（電話・メール）
- テクニカルサポートの連絡先
- 会社情報（会社名・住所・代表番号）

---

## PoC後の本番商談への移行フロー

1. **PoC終了2週間前**にレビューミーティングを設定する
2. 成功基準（KPI）の達成状況を定量的なデータで報告する
3. お客様のフィードバックをまとめ、課題・改善点を整理する
4. 本番契約に向けた提案書・見積書を準備して提出する
5. 本番移行の意思決定を確認し、契約手続きに進む

## 実装ガイド（AI向け指示）

このスキルが呼び出された際、AIは以下の手順で処理する。

1. ユーザーに企業名・課題内容・製品名・提案期間を確認する

3. 成功基準（KPI）は業種・課題に応じて具体的な数値例を提示する
4. 週次スケジュールは提案期間に合わせて展開する
5. 出力ファイル名を `POC-PROPOSAL-[会社名].md` とする
